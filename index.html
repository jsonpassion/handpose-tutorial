Chapter 2

컴퓨터는 손을 어떻게 볼까

같은 장면도 사람과 컴퓨터는 다르게 봅니다. 버튼을 눌러 컴퓨터의 시점으로 전환해 보세요.

사람의 눈. 하나의 "손"으로 인식합니다. 컴퓨터가 받는 입력은 다릅니다.

애플 플랫폼에서 이 관절 추출을 담당하는 것이 Vision 프레임워크입니다. 카메라 프레임을 입력하면 손에서 21개 관절(손목 1 + 손가락 4×5), 몸에서 19개 관절의 좌표를 반환합니다. 직접 구현할 필요 없이 iOS에 내장된 기능을 호출하면 됩니다.

직접 확인 — 내 손의 21개 관절

카메라 권한이 필요하며, 영상은 기기 밖으로 전송되지 않습니다. 이 데모는 브라우저용 MediaPipe로 동작합니다 — 애플 기기에서는 Vision 프레임워크가 같은 21개 관절을 찾습니다.

Chapter 3

픽셀이 아니라 관절을 학습한다

지하철 노선도는 건물과 골목을 지우고 역(점)과 노선(선)만 남긴 지도입니다. Vision의 관절 추출이 손에 하는 일이 같습니다 — 배경·조명·피부색을 지우고 점 21개와 그 연결만 남깁니다.

그래서 손 모양 분류기는 사진이 아니라 좌표 42개(21관절 × x·y)를 학습합니다. 배경을 바꿔 가며 두 방식의 확신도를 비교해 보세요.

이미지 분류 (픽셀 학습)98%

학습 데이터와 같은 초록 배경 — 자신만만합니다.

포즈 분류 (관절 학습)97%

관절 좌표는 배경과 무관 — 어디서든 같습니다.

수치는 실측이 아닌 개념 예시입니다. 방향은 실제와 같습니다 — 초록 배경 데이터로 학습한 포즈 분류기가 다른 환경에서 그대로 동작하는 이유입니다.

Chapter 5

Create ML 15분 코스 — 코드 없이 모델까지

Create ML은 Xcode에 포함되어 있습니다(Xcode → Open Developer Tool → Create ML). 폴더를 드래그하면 학습이 시작되고, 결과물은 앱에 바로 넣을 수 있는 .mlmodel 파일 하나입니다. 전 과정은 다섯 단계입니다.

클래스별 폴더에 사진 담기

폴더 이름이 곧 정답 라벨이 됩니다. 데이터 준비의 전부입니다.

HandPoseData/ ├── rock／ ✊ 726장 ├── paper／ ✋ 710장 ├── scissors／ ✌️ 752장 └── background／ ❓ 그 외 아무 손 (중요!)
background 클래스가 필요한 이유 — 없으면 모델은 모든 손을 가위·바위·보 중 하나로 분류합니다. "해당 없음"을 학습해야 불확실한 입력을 거를 수 있습니다.

Create ML에서 Hand Pose Classification 템플릿 선택

New Document → 템플릿 목록에서 Hand Pose Classification을 고릅니다. 손 액션·몸 액션도 같은 목록에 있습니다.

폴더를 드래그하고 Train

Training Data에 폴더를 놓고 Train을 누릅니다. 2,000장 규모면 수 분 내에 끝납니다. 내부적으로 모든 사진이 관절 21개 좌표로 변환된 뒤 학습됩니다.

Preview 탭에서 바로 테스트

학습이 끝나면 Preview 탭에서 카메라로 즉시 확인할 수 있습니다. Mac에서는 iPhone을 연속성 카메라로 연결하면 실시간으로 라벨이 표시됩니다.

확인 포인트 — 초록 배경 데이터로 학습한 모델이 다른 환경에서도 동작하는지. 동작한다면 3장의 "관절은 배경과 무관"을 눈으로 확인한 것입니다.

Get 탭 → .mlmodel 내보내기

Output에서 .mlmodel 파일을 저장하고 Xcode 프로젝트에 드래그합니다. 모델 파일은 수 MB 수준입니다. 앱 연동 코드는 6장에서 다룹니다.

Chapter 6

앱에 넣기 — 핵심은 20줄

카메라 프레임마다 Vision으로 관절을 추출하고, 그 좌표를 모델에 입력합니다. 핵심 흐름은 아래 코드가 전부입니다.

카메라 프레임AVFoundation
→

→
좌표 분류내가 만든 .mlmodel
→
앱 반응UI · 사운드 · 게임 로직
import Vision

let request = VNDetectHumanHandPoseRequest()
request.maximumHandCount = 1

func classify(_ frame: CVPixelBuffer) throws {
    let handler = VNImageRequestHandler(cvPixelBuffer: frame, orientation: .up)
    try handler.perform([request])
    guard let hand = request.results?.first else { return }  // 손이 없으면 조용히 패스

    // 관절 21개 좌표를 모델 입력으로
    let keypoints = try hand.keypointsMultiArray()
    let result = try model.prediction(poses: keypoints)

    // 확신도가 낮으면 무시
    if (result.labelProbabilities[result.label] ?? 0) > 0.85 {
        smoother.add(result.label)  // 최근 N프레임 다수결 버퍼
    }
}

입력 이름(poses)은 Create ML이 만들어 준 모델 그대로 씁니다. Xcode가 자동 생성한 클래스에서 확인할 수 있어요.

실전 품질을 좌우하는 것은 마지막 네 줄입니다. 확신도 임계값(0.85 미만 무시)과 프레임 다수결(최근 10프레임 중 과반 라벨만 채택) — 이 두 가지가 없으면 라이브 화면에서 라벨이 심하게 흔들립니다.

전체 앱 구조가 궁금하면 자료실의 애플 샘플 "Detecting Hand Poses with Vision"(핀치로 그림 그리기)이 가장 좋은 출발점입니다.

Chapter 9

더 깊은 질문들 — 만들 줄 아는 사람에서, 판단할 줄 아는 사람으로

기술을 돌리는 것과 기술로 좋은 제품을 만드는 것 사이에는 질문 몇 개가 있습니다. 팀 토론용으로 열어 보세요.

왜 하필 '관절'일까 — 사진을 통째로 학습하면 안 됐을까?

됩니다. 실제로 이미지 분류로 가위바위보를 풀 수도 있습니다. 하지만 비용이 다릅니다. 픽셀 6만 개에서 규칙을 찾으려면 배경·조명·피부톤의 온갖 조합을 커버할 만큼 데이터가 필요합니다. 관절 42개 숫자로 줄이면 "무엇이 손 모양을 결정하는가"만 남아 데이터가 수십 분의 일로 줄어듭니다.

대가도 있습니다. 관절 추출이 실패하면(저해상도, 장갑, 극단적 역광) 그 뒤는 전부 무너집니다. 100×100픽셀짜리 수어 데이터셋이 실패하는 이유가 바로 이것 — 파이프라인의 첫 단이 전체의 천장이 됩니다. "어떤 표현으로 압축해서 학습하는가"는 모든 머신러닝 설계의 첫 질문입니다.

모델이 "모르겠다"고 말하게 하려면?

모델은 기본적으로 거절을 모릅니다. 가위·바위·보만 배운 모델에 하트를 입력하면 셋 중 하나로 분류합니다. 그래서 두 가지 장치가 필요합니다: 학습 단계의 background 클래스("그 외"라는 정답지)와, 추론 단계의 확신도 임계값.

임계값을 얼마로 두느냐는 기술 문제가 아니라 제품 결정입니다. 리액션 캠의 오탐은 사소하지만, 수어 통역의 오탐은 주문을 틀리게 만듭니다. 오탐의 비용이 큰 앱일수록 임계값을 올리고, "인식하지 못함"을 정직하게 보여주는 UI를 설계해야 합니다.

데이터에 누가 들어 있는가?

HaGRID가 3만 명 이상의 인물을 모은 이유는 분명합니다 — 데이터에 없는 손은 인식되지 않기 때문입니다. 왼손잡이의 손, 어린이의 손, 어두운 조명의 손이 데이터에 없으면 그 사용자에게 이 앱은 동작하지 않는 제품입니다.

관절 기반 학습은 피부톤·배경 편향을 줄여 주지만 없애지는 못합니다 — 관절을 '추출하는' 모델 자체가 누군가의 데이터로 학습됐으니까요. 그리고 수어처럼 언어가 얽히면 편향은 더 근본적입니다: 한국수어는 미국수어와 다른 언어입니다. 케글의 ASL 데이터로 만든 앱은 한국 농인에게 무용지물입니다. "내 데이터셋에 누가 없는가"를 묻는 것이 좋은 ML 제품의 출발점입니다.

왜 애플은 전부 온디바이스로 돌릴까?

손 인식 앱은 카메라를 계속 켜 두는 앱입니다. 영상이 서버로 전송된다면 사용자가 신뢰하기 어렵습니다. 온디바이스는 성능 최적화이기 전에 신뢰 설계입니다 — 영상이 기기 밖으로 나가지 않는다는 사실이 카메라 상시 구동의 심리적 비용을 감당 가능하게 만듭니다.

함께 얻는 것: 오프라인 동작, 낮은 지연시간, 서버 비용 제로. 그리고 "영상은 기기를 떠나지 않습니다"라는 한 문장 — 사용자 신뢰에 가장 크게 기여하는 문장입니다.

터치보다 손짓이 나은 순간은 언제인가?

제스처가 이기는 조건은 대체로 넷입니다: 거리(화면이 멀 때 — TV, 무대), 점유(양손이 바쁠 때 — 요리, 운전, 커피 든 손목시계), 위생(만지면 안 될 때 — 수술실, 주방), 소음(말할 수 없을 때 — 도서관, 회의 중). 워치 더블 탭은 '점유'를, Vision Pro는 '거리'를 푼 사례입니다.

제스처가 불리한 조건도 있습니다: 장시간 사용 시 팔 피로(고릴라 암), 공공장소에서의 사회적 어색함. "사용자가 그 순간 왜 화면을 만질 수 없는가"에 답할 수 있다면 좋은 제스처 앱입니다.

포즈 다음은 무엇일까?

이 분야는 빠르게 확장되고 있습니다. Vision은 이미 3D 몸 포즈(깊이 포함)까지 제공하고, Vision Pro는 시선과 손을 조합해 "무엇을 보며 집었는가"를 읽습니다. 단일 입력(터치)에서 다중 신호(시선+손+음성)로 가는 흐름입니다.

오늘 배운 구조 — 신호를 압축하고(키포인트), 분류하고(모델), 불확실하면 거절한다(임계값) — 는 어디에나 동일하게 적용됩니다. 손이 아닌 다른 신호를 다루더라도 이 세 단계의 틀은 유효합니다.

Chapter 10 · 세션용

핸즈온 — 12분 만에 첫 모델

세션에서 함께 진행하는 실습입니다. 아래 준비물을 미리 확인하세요.

세션 전 준비물

  • Mac + Xcode 설치 — Create ML은 Xcode에 포함되어 있습니다 (Xcode → Open Developer Tool → Create ML)
  • 가위바위보 데이터셋 다운로드 — 위 데이터셋 지도의 케글 링크에서 zip을 받아 압축 해제 (세션장에서는 에어드롭으로도 공유합니다)
  • (선택) iPhone + 케이블 — 연속성 카메라로 연결하면 Preview 테스트가 수월합니다

0~2분 · 프로젝트 만들기

Create ML 실행 → New Document → Hand Pose Classification → 이름 짓고 저장.

2~4분 · 데이터 넣기

다운로드한 폴더(rock/paper/scissors)를 Training Data에 드래그. background 클래스는 시간이 남으면 각자 아무 손 사진을 몇십 장 추가해 실험해 봅니다.

4~9분 · Train

학습 중에는 진행 그래프 읽는 법, 데이터가 관절 좌표로 변환되는 과정, 데이터량에 따른 변화를 다룹니다.

9~12분 · Preview로 검증

카메라 앞에서 가위·바위·보 동작을 확인합니다. 초록 배경 데이터로 학습한 모델이 현장에서 동작하는 이유 — 오늘 원리의 핵심입니다.

잘 안 될 때

  • Preview가 반응 없음 — 손이 프레임에 충분히 크게(화면의 1/4 이상) 들어오게 가까이. 역광이면 방향을 바꿔 보세요.
  • 라벨이 심하게 떨림 — 정상입니다. 앱에서는 확신도 임계값 + 프레임 다수결로 잡습니다 (6장 코드 참고).
  • 엉뚱한 손도 가위바위보로 인식 — background 클래스가 없기 때문입니다. 9장 두 번째 질문 참고.
  • 학습이 너무 느림 — 다른 앱을 닫고 전원 연결. 그래도 느리면 클래스당 200장으로 줄여도 데모 품질은 충분합니다.

이번 주 팀 미션

오늘 만든 모델을 확장하거나 데이터셋 지도에서 새로 선택해, 손 또는 몸으로 조작하는 앱 하나를 다음 미팅 팀 공유 시간에 시연합니다. 모델은 Create ML로 15분이면 충분합니다 — 나머지는 각자의 바이브 코딩으로.
